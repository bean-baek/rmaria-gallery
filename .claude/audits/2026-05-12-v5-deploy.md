## Warnings (non-blocking but document)

- **`<html lang="ko">`** — dist/index.html declares Korean as document language, but page chrome and meta strings are mixed EN/KO. This is intentional given the dual-language captions, but verify the OG snippet renders cleanly on Twitter/Facebook validators after deploy.
- **`og:locale = ko_KR`** matches the lang attribute (consistent). If you target English-first audiences add `og:locale:alternate` later.
- **`site.webmanifest`** uses absolute paths (`/favicon.svg`) while `index.html` uses relative (`./favicon.svg`). At root they resolve the same; document as a minor inconsistency.
- **`scripts/optimize-images.mjs`** is referenced by `build:full` but the script's existence wasn't validated against required input dir `r.maria/src/`. CI must have `r.maria/src/` present (or the build must use plain `npm run build` after a one-time image generation).
- **Sitemap uses hash-fragment URLs** — the file itself notes "Hash fragment URLs may not be fully crawlable; we publish them as a hint until SSG is added." Acceptable for v5; ensure GSC submission expectations are tempered.
- **`<link rel="manifest" href="./site.webmanifest">`** — relative path resolves at root but may 404 on deep routes if served by static host without SPA rewrite. Cloudflare Pages serves the file at `/site.webmanifest` so this is fine after deploy.

## Deployment Readiness Blockers

1. **NOT A GIT REPOSITORY.** `git rev-parse --is-inside-work-tree` → `fatal: not a git repository`. **You must `git init`, commit the project, push to GitHub/GitLab, then connect the repo to Cloudflare Pages.** Cloudflare's Git Integration is the standard ship path; without it you'd have to switch to Direct Upload (Wrangler) for every deploy.

2. **`r.maria/src/` is NOT in `.gitignore`** — good. The current `.gitignore` is:
   ```
   node_modules/
   dist/
   public/works/
   .DS_Store
   *.log
   ```
   So CI *can* regenerate `public/works/` from `r.maria/src/` via `npm run build:full`. BUT — Cloudflare's default Pages build command is `npm run build`, NOT `build:full`. Either (a) set the Pages build command to `npm run build:full`, or (b) commit `public/works/` (remove from `.gitignore`).

3. **Environment variables to configure in Cloudflare Pages dashboard:**
   - `INQUIRY_EMAIL` (required) — destination address for inquiries; without it `/api/inquire` returns 500.
   - `MAIL_FROM_DOMAIN` (optional) — defaults to `rmaria.pages.dev`. Set this only if you switch to a custom domain.
   - `NODE_VERSION=22` (recommended) — `sharp@^0.33.5` and Vite 6 work best on Node 22.

4. **MailChannels caveat.** As of mid-2024 MailChannels began requiring sender-domain DNS auth (SPF/DKIM `_mailchannels` TXT record) for *all* customers, including Cloudflare Workers/Pages. The `pages.dev` subdomain inherits Cloudflare's existing setup so it *should* still work, but if mail delivery fails after deploy verify the MailChannels domain-lockdown record. If you ever move to a custom domain you must add the lockdown TXT yourself.

5. **Pages Functions discovery.** Confirm Cloudflare Pages picks up `functions/api/inquire.js`. The conventional path (`functions/<route>`) is correct — `/api/inquire` will route to `onRequestPost`. No `wrangler.toml` needed for the default Pages setup.
