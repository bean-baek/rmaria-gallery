## Warnings (non-blocking but important)

### W1 — JS gzipped bundle exceeds user comfort range by ~10 KB
- **Evidence**: `dist/assets/index-BeYmg0WF.js` is **511.47 KB raw / 130.88 KB gzipped**. Original plan target was ≤60 KB gzipped; user-accepted range is 80–120 KB gzipped. We are at 130.88 KB — **70.88 KB over the original plan target**, and **~10.88 KB above the upper end of the user's stated comfort zone**.
- **Cause**: Three.js core (`WebGLRenderer`, `Scene`, `OrthographicCamera`, `PlaneGeometry`, `ShaderMaterial`, `Mesh`) pulls in WebGLRenderer internals that cannot be tree-shaken further. The plan underestimated this footprint.
- **Functional impact**: None — Vite reports 0 errors, chunk-size warning is informational only.
- **Recommendation**: See Recommendations section. Either accept the new bar (~130 KB is still fast on modern networks) or replace Three.js with a hand-rolled raw-WebGL grain renderer (likely <5 KB), which would bring the bundle back under ~70 KB gzipped. This is a user-level product decision.

## Recommendations

The single warning is the bundle-size gap. Two viable paths, both decisions for the user:

**Option A — Accept the 130 KB gzipped bar.**
- Pros: zero refactor work; Three.js is a known quantity and the grain renderer is genuinely small/clean (~140 LoC).
- Cons: ships ~10 KB over the user's previously-stated comfort zone; first-load JS parse cost is real on low-end mobile.
- If chosen: update the layout/three-js policy documents to record 130 KB gzipped as the new agreed bar, so future audits do not re-flag.

**Option B — Replace Three.js with raw WebGL for the grain.**
- Scope: ~80–120 lines hand-written WebGL — compile vertex/fragment shaders, set up a single full-screen quad, run the same RAF loop. The shader source already in `grain.js` ports unchanged.
- Expected delta: JS gzipped should drop from ~130 KB to **well under 30 KB** (Three.js is essentially the entire bundle today).
- Risk: small — the grain renderer is the only Three.js consumer, so there's no second usage to refactor.
- Effort: ~1–2 hours including disposer parity.

**Recommendation**: if the bundle-size budget matters to the user beyond the comfort range, Option B is high-leverage and low-risk. If not, lock Option A in and update the policy doc to match reality. Either way, the v3 changeset itself is ship-ready.
